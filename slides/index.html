Lo que nos pasa

El costo

Background Agents

Agentes de IA que trabajan en segundo plano, de forma autónoma.

  • Reciben un issue o tarea
  • Ejecutan en una VM en la nube
  • Trabajan en una rama separada
  • Crean un PR cuando terminan
  • No interrumpen tu flujo de trabajo

"Dale la tarea al agente, seguí trabajando, revisá el PR después."

Flujo de trabajo típico

  1. Crear issue en GitHub con descripción clara
  2. Asignar al Background Agent desde Cursor
  3. Seguir trabajando en lo tuyo
  4. Recibir notificación cuando el agente termina
  5. Revisar el PR y hacer merge

Tiempo típico: 2-10 minutos dependiendo de la complejidad

Casos de uso ideales

Funciona bien

  • Bug fixes bien definidos
  • Agregar tests
  • Actualizar documentación
  • Refactors acotados
  • Agregar validaciones
  • Tareas con specs claros

No ideal (aún)

  • Features nuevas complejas
  • Cambios arquitecturales
  • Tareas que requieren contexto de negocio profundo
  • Integraciones con servicios externos

Limitaciones y consideraciones

  • Requiere buena descripción — cuanto mejor el issue, mejor el resultado
  • No reemplaza code review — siempre revisar el PR
  • Funciona mejor con tests — el agente los usa para validar
  • Acceso a secrets limitado — por seguridad, no tiene acceso a .env
  • Feature en beta — sigue mejorando

Tips y mejores prácticas

  • Escribí issues claros y específicos — el agente es tan bueno como el issue
  • Incluí tests en el repo — el agente los corre para validar
  • Empezá con tareas chicas — bugs, docs, validaciones
  • Siempre revisá el PR — no es merge automático
  • Usá .cursorrules — para darle contexto y estilo al agente
  • Iterá — podés comentar en el PR para que el agente ajuste
